**Fact:** The CD control-flow design was settled in a July 2026 design review (recorded in `cd-initiative-implementation-plan-2026-07.md` §3.6 and `cicd-pipeline-diagram-2026-07.html`):

- **Two one-button human decisions, everything else automatic.** `detect_charts` stays manual (picks *which MR* enters the pipeline — deliberate prioritization when several MRs are open). `bump:<site>` stays manual (THE go-live decision — which site, when). CRM-763's "no manual step" acceptance criterion is re-scoped to "no manual *work*, only one-button manual *decisions*" — permanent, revisit only if requirements change. **Both buttons are Maintainer-gated via an in-script members-API check** (`$GITLAB_USER_ID` of the clicker resolved via `/projects/:id/members/all/`, fail unless `access_level >= 40`) — plan §3.7 has the snippet. Protected Environments was the first choice but requires Premium; **gitlab.expertflow.com verified Free tier 2026-07-10** (no "Protected environments" section in cim-solution's Settings → CI/CD at full Maintainer view). Upgrade path if the instance is ever licensed: switch to Protected Environments. Gating becomes load-bearing when the factory moves into `cim-solution` (every stream dev is a member there).
- **`cx-environments-cd` is the complete, self-contained state of every site.** Charts by version pin (package lives in the Helm registry); pre/post-deployment resources as **actual files copied into `sites/<site>/`** by bump (no registry exists for them — a SHA reference was explicitly rejected in favor of real files). One bump changeset advances site level + ALL tenant folders atomically.
- **Commit-as-trigger:** any commit on `cx-environments-cd` main deploys what it touched (`rules: changes: sites/<site>/**`); `[skip ci]` opts out. Direct config edits by engineers are a first-class deploy path. A pipeline-trigger-API mechanism was proposed and rejected in favor of this.
- **New-chart onboarding:** if a bump changeset includes a chart not yet pinned on the site, the *entire* changeset goes into one auto-generated MR (pin + values seeded verbatim from the chart's `values.yaml`; reviewer edits site-specifics; merge = the single deploy moment for everything). Never split "existing charts now, new chart later." This was an explicit re-litigation of the no-fallback-values decision — resolved as review-once-then-flows.
- **Unified skip-if-unchanged:** per-site in-cluster marker holds the last-applied `cx-environments-cd` commit SHA; each run diffs `marker..HEAD -- sites/<site>/` and touches only what changed. Marker advances only on full success; rollback resets it.
- **Rollback reverts state, not just cluster:** `helm rollback` + `[skip ci]` commit reverting the pins/files. Cluster-only rollback re-arms the failure on the next deploy.
- **Process assumption (not enforced):** two MRs touching the same pre/post-deploy file never deploy concurrently — team rule (deploy one, merge to RC, second MR pulls RC first). Pipeline tripwire: bump warns loudly when overwriting a `sites/` file modified since the last bump.

**History of this decision (why earlier records disagree):** the original plan/diagram said "no manual step at all"; an interim correction (since superseded) put the manual gate on `pre-deploy:<site>`, reasoning bump was automatic — that was a misreading of bot-authored commits (the POC's `bump:<site>` in `tmp/cx-charts-cd/.gitlab-ci.yml` is a manual per-team button; the bot identity is just CI's git author). The factory repo (detect/package/publish/bump) is `cx-charts-cd` in the POC and moves into `cim-solution` in the target design.

**How to apply:** When implementing Phase 2+ (`.gitlab-ci.yml` changes in either repo), the manual gates go on `detect_charts` and `bump:<site>` only — pre-deploy/deploy/post-deploy/test/notify have no `when: manual`. If a doc or ticket still says "fully automatic, no human click" (objective-realignment doc's "no person having clicked anything", CRM-763's original AC) or "gate on pre-deploy", those predate this decision — plan §3.6 wins.
