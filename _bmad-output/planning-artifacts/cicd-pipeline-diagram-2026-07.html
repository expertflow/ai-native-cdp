CD Initiative — Revised Design, July 2026

The complete release pipeline

From a merge request on cim-solution to a deployed, tested, and — if broken — automatically reverted release. Three repositories, one continuous flow, no manual step in the middle.

cim-solution — chart lifecycle cx-environments-cd — deploy lifecycle playwright-automation-script — test logic (included, not duplicated) pass path fail → auto-revert human action point

cim-solutionchart lifecycle — triggered by the MR

  1. detect_charts Stream team's MR lands. Diffs against the target branch to find which Helm charts changed.
  2. package_charts Auto-calculates the next -rc.N version from the registry, updates Chart.yaml, runs helm lint, packages the chart.
  3. publish_charts Pushes the packaged chart to the GitLab Helm registry — the single source of truth for chart versions.
  4. bump:<site>manual One button per site. The only human click on this side of the pipeline — a site pulls the new version when it's ready for them, on their own schedule.
writes version pin → <site>/apps.yaml →

cx-environments-cddeploy lifecycle — one pipeline, per site (and per tenant on multi-tenant sites), fully independent

  1. pre-deploy:<site> ConfigMaps, TLS secrets, ImagePullSecrets, Vault/CSI RBAC — templated per environment, applied before Helm touches anything.
  2. deploy:<site> helm upgrade -f <site>/values.yaml — this site's config, not the chart's defaults. Inline rollback if the upgrade itself fails.
  3. post-deploy:<site> Tenant creation, Grafana dashboards, Metabase, MinIO data — idempotent, skipped if already applied.
  4. test:<site>extends Runs the regression suite in this pipeline — the job definition is included from playwright-automation-script, not cloned or duplicated. See the card at right.

— straight through No rollback job runs. Its absence is itself the signal.
Tests failed
rollback:<site> helm rollback to the previous known-good revision.
notify:<site> Always runs. Reads test:<site>'s own results directly — same pipeline, no cross-project lookup — and whether rollback fired. Posts one aggregate Google Chat card either way.

playwright-automation-script

Owns the test script, browser image, and reporter config — QA's repo, QA's cadence. include: project: 'qa/playwright-automation-script', ref: 'v5.4.0' pulls the job definition into cx-environments-cd at the pinned version; nothing here is cloned or re-typed elsewhere.

Its own .gitlab-ci.yml is untouched by any of this — QA still runs it standalone for authoring and debugging, exactly as before.

!allow_failure must stay false on test:<site> — a masked failure means rollback never fires.
!rollback stays when: manual until the suite's flakiness sources are fixed (CRM-777) — auto-revert on a flaky test is worse than no rollback.
!Migration-carrying releases skip auto-rollback and route to manual approval instead (CRM-780) — helm rollback doesn't undo a database migration.
!Multi-tenant sites add a tenant layer — pre-deploy-tenant/deploy-tenant:<site>:<tenant> jobs create a per-tenant namespace, copy in certs/secrets from ef-external/expertflow/vault, and deploy MTT-single — same test → rollback → notify shape shown above, just parameterized per tenant instead of per site. Single-tenant sites (devops, RMT) skip this layer entirely. See the CD plan §2 for the full namespace model.
